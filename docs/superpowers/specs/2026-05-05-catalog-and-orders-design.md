# Catalog & Orders Design

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship a minimum viable catalog (products + batches + COA PDFs) and a fully functional orders system (kanban, order detail, create from Orders page and from Inbox) as one end-to-end feature.

**Architecture:** Flat product model with a `product_family` grouping field, one-to-many batches per product, orders referencing products and batches via line items. Order creation works from two entry points — a modal on the Orders page and a right-rail panel inside the Inbox — sharing one form component. All mutations are server actions; data fetching is server-side; one lightweight client API route powers the product picker.

**Tech Stack:** Next.js 15 App Router, Supabase (PostgreSQL + RLS + Storage), TypeScript, existing `pt-*` CSS design system.

## Data Model

| Column | Type | Notes |
|---|---|---|

## Storage

COA PDFs stored in a `coa` Supabase Storage bucket (private, RLS via JWT tenant_id). Path convention: `{tenantId}/{batchNumber}.pdf`. Served via signed URLs (1-hour expiry) generated on demand — no permanent public URLs.

## UI Components

### Catalog page (`/catalog`)
- Product list grouped by `product_family`, sorted alphabetically within each family.
- Each product row shows SKU, name, unit price, batch count, total stock. Low-stock rows highlighted.
- Expandable per-product section lists batches: batch number, stock, expiry, COA download link (signed URL opens in new tab).
- **Add product**: inline form — SKU, name, product family, unit price, description. Submits via `createProduct` server action.
- **Add batch**: per-product button opens small inline form — batch number, stock count, expiry date, COA PDF file picker. Submits via `createBatch` server action which returns a signed upload URL, then uploads the file client-side directly to Supabase Storage.

### Orders kanban (`/orders`)
- Existing drag-and-drop kanban UI wired to real `orders` data.
- Five columns: Awaiting payment · Confirming · Packing · Shipped · Delivered.
- Dragging a card calls `updateOrderStatus` — updates DB and appends an order event.
- Each card shows: ref number, customer name, channel icon, total amount, time since created.
- **"New order" button** in page header opens `CreateOrderModal`.

### Order detail page (`/orders/[orderId]`)
New page implementing the existing `order-detail.jsx` design:
- **Header**: ref number, status pill, channel, customer name, back button, Message and Print label actions.
- **Stepper**: 5-step progress bar reflecting current status.
- **Line items table**: SKU · Item · Batch · COA link · Qty · Unit price · Line total. Subtotal / Shipping / Total footer.
- **Payment card**: asset, amount, receiving address, tx hash (once confirmed), confirmation progress bar for `confirming` status.
- **Shipping card**: address (masked toggle), carrier, tracking, ETA, packaging notes.
- **Operator notes**: textarea, auto-saves on blur via `saveOrderNotes`.
- **Right rail**: customer card (links to `/customers/{id}`), activity timeline (order events), chat excerpt (last 3 messages from linked conversation, links through to Inbox).

### Create order form (`CreateOrderForm`)
Shared component used in two containers:

**Modal container** (from Orders page):
- Customer search/select at top (searches `customers` by name or handle).
- Line items: product family search → selects SKU/dosage → auto-fills unit price → qty input. Add/remove rows.
- Payment method selector (USDT, BTC, Cash, other).
- Shipping address fields.
- Operator notes.
- Submit calls `createOrder` server action, redirects to new order detail page.

**Right rail container** (from Inbox):
- "Create order" button added to conversation header (alongside Snooze and Mark done).
- Clicking it swaps the right rail to `CreateOrderForm` with customer pre-filled and `conversation_id` captured.
- Conversation stream remains fully visible — operator can scroll messages to reference address, items requested.
- On submit, rail snaps back to customer view. A success toast shows the new ref number.

## Server Actions

All in `src/app/catalog/actions.ts` and `src/app/orders/actions.ts`.

**Catalog:**
- `createProduct(data)` — validates SKU uniqueness per tenant, inserts into `products`.
- `createBatch(data)` — inserts into `batches`, returns a Supabase Storage signed upload URL for the COA PDF.

**Orders:**
- `createOrder(data)` — inserts `orders` row (triggers ref_number generation), inserts `order_items` rows, inserts first `order_events` row (`"Order drafted from chat"` if `conversation_id` present, else `"Order created"`). Returns `{ orderId, refNumber }`.
- `updateOrderStatus(orderId, status)` — updates `orders.status`, inserts order event `"Moved to {status}"` with actor `operator`.
- `updateOrderShipping(orderId, data)` — updates shipping fields.
- `saveOrderNotes(orderId, notes)` — updates `orders.notes`.

## API Route

**`GET /api/catalog/products?q={query}`** — lightweight JSON endpoint for the create-order product picker. Searches `products` by `sku`, `name`, or `product_family` for the authenticated tenant. Returns `[{ id, sku, name, productFamily, unitPrice }]`. Used client-side as the operator types.

## RLS Policies

All five tables get the same pattern already used across the codebase:
```sql
USING (tenant_id = auth_tenant_id())
WITH CHECK (tenant_id = auth_tenant_id())
```

COA storage bucket gets a policy allowing read/write only where the storage path starts with the user's `tenant_id`.

## What This Does Not Include

- Automated status advancement (e.g. auto-move to Packing on payment confirmation) — that belongs to the Automations feature.
- Broadcast audience segments derived from orders — that belongs to the Broadcasts feature.
- Real blockchain tx monitoring — tx hash is entered manually by the operator for now.
- Print label functionality — button present in UI but no-op for now.
